# Contract Definitions

## SME OS Input Contract (Conceptual)

SME OS accepts generic business metrics that represent:

### Financial Inputs
- Cash flow data (inflows, outflows, timing)
- Financial position (current balance, projected balance)
- Revenue streams (amount, frequency, variability)
- Cost structures (fixed, variable, timing)

### Operational Inputs
- Resource capacity (units available, time periods)
- Resource utilization (current usage, historical patterns)
- Time-based data (periods, dates, durations)
- Constraints (limits, rules, boundaries)

### Context Inputs
- Business context (size, maturity, industry phase)
- Historical patterns (trends, seasonality, cycles)
- External factors (market conditions, economic indicators)
- Decision history (past decisions and outcomes)

### Input Characteristics
- **Generic**: No vertical-specific terminology
- **Normalized**: Data is standardized and validated
- **Time-bound**: All inputs are scoped to specific time periods
- **Quantitative**: Primarily numeric data, minimal categorical

## SME OS Output Contract (Conceptual)

SME OS returns:

### Alerts
- Alert type (risk, opportunity, anomaly)
- Severity level (critical, warning, informational)
- Time relevance (immediate, near-term, future)
- Affected domain (cash, risk, labor, forecast)
- Alert message (generic explanation)

### Confidence Scores
- Decision confidence (0-1 scale)
- Data quality score (how reliable are inputs)
- Model certainty (how certain is the evaluation)
- Recommendation strength (how strongly should this be considered)

### Explanations
- Decision reasoning (why this alert was generated)
- Contributing factors (what inputs led to this)
- Context (what conditions apply)
- Implications (what this means)

### Recommendations (if applicable)
- Suggested actions (generic, not prescriptive)
- Alternative scenarios (what-if possibilities)
- Trade-offs (costs and benefits)

### Output Characteristics
- **Non-prescriptive**: Alerts inform, they don't command
- **Generic**: No vertical-specific language
- **Authoritative**: Calm, conservative tone
- **Sparse**: System speaks rarely, but matters when it does

## Alert Contract (Conceptual)

Alerts are notifications, not commands:

### Alert Structure
- **ID**: Unique identifier
- **Type**: Risk, opportunity, anomaly, threshold
- **Severity**: Critical, warning, informational
- **Domain**: Cash, risk, labor, forecast
- **Timestamp**: When alert was generated
- **Time Horizon**: Immediate, near-term (1-7 days), medium-term (1-4 weeks), long-term (1+ months)
- **Message**: Generic explanation
- **Confidence**: How certain is this alert
- **Contributing Factors**: What led to this alert
- **Context**: Relevant conditions

### Alert Behavior
- **Informational**: Alerts inform, they don't prescribe actions
- **Non-intrusive**: Alerts don't interrupt workflows unnecessarily
- **Contextual**: Alerts include enough context to understand
- **Actionable**: Alerts provide enough information for user to decide

### Alert Lifecycle
- Generated by SME OS when thresholds are crossed
- Sent to Hospitality AI via output contract
- Displayed in Hospitality AI UI
- User acknowledges or dismisses
- Alert history stored for decision memory
